# A-Maze-ing

## Description

**A-Maze-ing** is a Python maze generator and terminal visualizer built for the 42 curriculum. The program reads a configuration file, generates a valid maze, writes it to an output file using the required hexadecimal wall encoding, computes the shortest path between the entry and exit, and provides an interactive terminal rendering of the result. The maze also tries to embed a visible **"42" pattern** made of fully closed protected cells when the maze size or the entry/exit coordinates allows it.

The project is organized around reusable core classes:
- `Parsing`: parses and validates the configuration file.
- `Cell`: represents one maze cell and its four walls.
- `Maze`: stores the full maze, validation logic, path-finding, the 42 pattern, and output serialization.
- `MazeGenerator`: generates the maze structure while preserving protected cells.
- `MazeRenderer`: displays the maze in the terminal and provides interactive controls.

## Features

- Configuration-file driven generation
- Perfect or imperfect maze generation
- Automatic shortest-path computation
- Validation of maze coherence and connectivity
- Hexadecimal output format required by the subject
- Terminal visualization with colors
- Interactive controls to regenerate the maze, toggle the path, change colors, and quit
- Reusable Python classes for future projects

## How it works

### 1. Configuration parsing

The program starts by reading a text configuration file with one `KEY=VALUE` pair per line. Empty lines and lines beginning with `#` are ignored. The parser validates all mandatory fields and raises a clear `ConfigError` when a value is missing or invalid. The current implementation enforces:
- `WIDTH` and `HEIGHT` as integers
- `ENTRY` and `EXIT` inside maze bounds (integers)
- `PERFECT` as a boolean-like value
- `OUTPUT_FILE` path to the output file as a string
- optional `SEED` as an integer
- any unknown keys are stored in `extra` for later use

### 2. Maze construction

A `Maze` instance is created from the validated configuration. Each cell starts with all four walls closed. The maze object also exposes helpers to:
- access cells and neighbors
- remove or restore walls between adjacent cells
- validate coherence between neighboring walls
- ensure there are no isolated cells
- ensure there is no forbidden 3x3 open area
- compute the shortest path with breadth-first search
- export the maze to the subject output format

### 3. The "42" pattern

Before generation, the maze attempts to place a centered **42 pattern** using protected cells. These cells remain fully closed so the pattern stays visible in the final rendering. If the maze is too small, or if the pattern would overlap the entry or exit, it is skipped and a warning is printed. The minimum supported size for the pattern in the current implementation is **8x7**.

### 4. Generation

The maze is generated by `MazeGenerator`, which performs a randomized depth-first traversal from the entry cell. It recursively removes walls between cells while preserving the 42 pattern and rejecting openings that would create a forbidden 3x3 open area. If the maze is configured as imperfect, the generator then opens some additional walls to create multiple valid paths.

### 5. Validation and export

Once generation is finished, the maze is validated to ensure it satisfies the subject constraints:
- border walls are coherent
- neighboring walls match
- no 3x3 open area exists
- all non-protected cells are reachable
- a valid path exists between entry and exit

If valid, the maze is written to the configured output file. The main program also starts the terminal renderer for interactive visualization.

## Main generation algorithm

The chosen generation algorithm is a **randomized backtracker**, which is a depth-first search based maze generation algorithm.

### Principle

1. Start from an initial cell (top-left) and push it to the stack.
2. Mark the current cell as visited.
3. While the stack is not empty, look at the cell at the top of the stack.
4. Find all valid, unvisited neighboring cells.
5. If there is at least one unvisited neighbor:
   - Randomly choose one.
   - Remove the wall between the current cell and the chosen neighbor.
   - Mark the neighbor as visited and push it to the stack (it becomes the new current cell).
6. If there are no unvisited neighbors available, pop the cell from the stack to backtrack to the previous one.
7. **Imperfection phase:** If `PERFECT=False`, randomly break ~20% of internal walls to create loops, while strictly preventing any forbidden 3x3 open areas.

### Why this algorithm

We chose this algorithm because it is:
- simple to implement and explain,
- efficient for grid mazes,
- well suited for building perfect mazes,
- easy to extend with extra constraints like protected cells and forbidden open areas.

It also matches the project requirements well because it naturally produces a connected maze and can be adapted for imperfect mazes by opening additional passages afterward.

## Bonus generation algorithm

The chosen generation algorithm is a **randomized recursive backtracker**, which is a depth-first search based maze generation algorithm.

### Principle

1. Start from the entry cell.
2. Mark the current cell as visited.
3. Randomly shuffle the four cardinal directions.
4. For each valid unvisited neighbor:
   - remove the wall between the current cell and the neighbor,
   - reject that opening if it creates a forbidden 3x3 open area,
   - recurse into the neighbor.
5. If the maze is not perfect, open additional internal walls afterward to create extra paths.

### Why this algorithm

We chose this algorithm because it is:
- simple to implement and explain,
- efficient for grid mazes,
- well suited for building perfect mazes,
- easy to extend with extra constraints like protected cells and forbidden open areas.

It also matches the project requirements well because it naturally produces a connected maze and can be adapted for imperfect mazes by opening additional passages afterward.

## Path-finding algorithm

The shortest path is computed with **breadth-first search (BFS)**. Starting from the entry, the maze explores reachable neighbors layer by layer until the exit is found. The recorded direction history forms the shortest valid path as a string containing only `N`, `E`, `S`, and `W`.

### Why BFS
We chose BFS for path-finding because it is:

guaranteed to find the shortest path in an unweighted grid maze, which fully matches our requirement to compute "the shortest path between the entry and exit",
simple and efficient with O(width × height) time complexity and O(width × height) space complexity, which is optimal for this problem,
intuitive to implement and verify, with clear layer-by-layer exploration that is easy to debug and understand,
naturally compatible with our grid-based maze structure where all moves (N, E, S, W) have equal cost,
appropriate for the project scope, as more complex algorithms like A* or Dijkstra would add unnecessary overhead without benefit since all distances are uniform.
BFS is the standard and most suitable algorithm for finding shortest paths in unweighted mazes, making it an excellent fit for this implementation.

## Output file format

The maze is written row by row using **one hexadecimal digit per cell**. Each bit represents whether a wall is closed:

| Bit | Direction |
|---|---|
| 0 (LSB) | North |
| 1 | East |
| 2 | South |
| 3 | West |

A closed wall is encoded as `1`, and an open wall as `0`. In the code, the cell converts its four walls to hex using the order `W S E N`, which corresponds to bits 3, 2, 1, 0. For example, `A` (`1010` in binary) means west and east are closed. After the maze grid, the output contains a blank line, then the entry coordinates, exit coordinates, and the shortest path.

## Visual representation
The subject requires a visual display that shows walls, entry, exit, and the solution path. This project uses a **terminal ASCII renderer** with ANSI colors. The renderer displays:

### Main Visualizer

- `🟥` for the entry,
- `🟪` for the exit,
- `⬛` for protected cells of the 42 pattern,
- `🟦` for cells belonging to the shortest path when enabled.

### Bonus Visualizer

- `E` for the entry,
- `X` for the exit,
- `█` for protected cells of the 42 pattern,
- `●` for cells belonging to the shortest path when enabled.

### Interactive controls

The current renderer supports the following keys:

#### Main Visualizer

| Key | Action |
|---|---|
| `1` | Regenerate a new maze |
| `2` | Show/Hide path from entry to exit |
| `3` | Rotate maze colors |
| `4` | Export maze in .svg |
| `5` | Show/Hide maze animation |
| `6` | Quit the program |

#### Bonus Visualizer

| Key | Action |
|---|---|
| `r` | Regenerate a new maze |
| `p` | Show or hide the shortest path |
| `c` | Change the color theme |
| `q` | Quit the program |

## Reusable code

A major goal of the project is to keep the maze logic reusable for a future project. The most reusable part of this codebase is the combination of:
- `Cell`
- `Maze`
- `MazeGenerator`
- `Parsing`

Together, these classes let you parse a config, build a maze, generate it, validate it, inspect its cells, compute a path, and export it. This matches the reusability requirement of the subject, which asks for a standalone reusable generator module.

### Minimal example from the current codebase

```python
from mazegen import Parsing, Maze, MazeGenerator

parser = Parsing("config.txt")
parser.parse()

maze = Maze(parser)
maze.apply_42_pattern()

generator = MazeGenerator(maze)
generator.generate()

print(maze.to_hex_output())
print(maze.find_shortest_path(maze.entry, maze.exit))
```

### What can be reused

- the config parser and validation rules,
- the maze model and wall manipulation methods,
- the generation algorithm,
- the path-finding method,
- the output serializer,
- the terminal renderer if a CLI display is needed.


## Error handling

The code is structured to fail gracefully:
- `Parsing` raises `ConfigError` for invalid configuration files.
- The main entry point raises `MazeError` for generation or output failures.
- File writing errors are caught and reported clearly.
- The main program prints a readable message instead of crashing unexpectedly.

## Technical choices

### Python and typing

The subject requires Python 3.10+, flake8 compliance, type hints, and mypy compatibility. The uploaded code follows that overall direction with type annotations and docstrings in the main modules.

### Terminal rendering over MLX

The subject allows either terminal ASCII or MiniLibX. This implementation uses the terminal approach, which keeps the project lightweight and easy to run on any standard environment without graphical dependencies.
